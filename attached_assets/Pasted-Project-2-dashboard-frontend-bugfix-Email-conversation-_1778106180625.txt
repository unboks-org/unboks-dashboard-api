Bug:
Email conversations appear in the inbox list, but when clicking an Email conversation, the detail pane does not load the message history.
The right side stays blank/loading.

WhatsApp conversations appear to load correctly.

Likely cause:
The Python backend uses the `phone` field as the conversation identifier.
For email conversations, the identifier can contain special characters, for example:
email::subj:workspace-noreply@google.com:boost productivity...

If the frontend builds the detail URL without encodeURIComponent, the request can break or hit the wrong route.

Confirmed API list shape includes:
{
  "phone": "email::subj:workspace-noreply@google.com:...",
  "customer_name": "...",
  "channel": "email"
}

Required behavior:
1. Clicking an Email conversation must load the full message thread.
2. Clicking WhatsApp conversations must still work.
3. The frontend must safely encode the conversation identifier when calling:
   GET /api/{client}/dashboard/api/messages/conversations/{phone}
4. Use encodeURIComponent on the id/phone/external conversation key before building the URL.
5. Do not double-encode if the code already encodes somewhere else.
6. Detail pane should show a calm error state if the API fails, not an empty blank pane.
7. Preserve email display name, badge, timestamp, and list behavior.
8. Do not change backend/API.
9. Do not guess a different id. Use the backend-provided conversation identifier.
