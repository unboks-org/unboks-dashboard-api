BUG:
Settings → Escalation Alerts currently saves only to browser localStorage.

Investigation confirmed:
artifacts/unboks/src/hooks/use-escalation-notification-preferences.ts uses:

const STORAGE_KEY = "unboks_escalation_notify_prefs"

and save() only does:

localStorage.setItem(...)

This means alert destinations never reach the backend.
So backend cannot send escalation alerts to the configured WhatsApp / Telegram / Messenger destinations.

Jr has shipped backend endpoints:
GET /api/{client}/dashboard/api/settings/escalation-alerts
PUT /api/{client}/dashboard/api/settings/escalation-alerts

Frontend must wire to these real endpoints.

Required behavior:
1. On Settings load, fetch escalation alert settings from backend.
2. Populate the UI from backend settings.
3. When user clicks Save changes, PUT settings to backend.
4. Keep localStorage only as a fallback/cache, not the source of truth.
5. If backend fetch fails because endpoint missing/unavailable, use localStorage fallback and show calm note.
6. If backend save succeeds, show Saved.
7. If backend save fails, show backend error and do not pretend backend saved it.
8. Do not silently fail.
9. Keep existing UI fields:
   - Email always on
   - WhatsApp toggle + destination
   - Messenger toggle + destination
   - Telegram toggle + destination

Backend request shape:
PUT /settings/escalation-alerts

Body:
{
  "channels": {
    "email": {
      "enabled": true,
      "destination": "default"
    },
    "whatsapp": {
      "enabled": true,
      "destination": "+351963618003"
    },
    "messenger": {
      "enabled": false,
      "destination": ""
    },
    "telegram": {
      "enabled": true,
      "destination": "@calvinadamus"
    }
  }
}

Expected GET response shape:
{
  "channels": {
    "email": {
      "enabled": true,
      "destination": "default"
    },
    "whatsapp": {
      "enabled": true,
      "destination": "+351963618003"
    },
    "messenger": {
      "enabled": false,
      "destination": ""
    },
    "telegram": {
      "enabled": true,
      "destination": "@calvinadamus"
    }
  }
}

Also support if backend returns a flat shape, but preferred is channels.

- Email + WhatsApp have real delivery
- Telegram + Messenger currently record skipped/provider not configured until providers are connected

So UI should not imply Telegram/Messenger are actively delivering if backend reports skipped/provider_not_configured.

If backend returns deliveryStatus per channel, show a small badge:

If backend does not return deliveryStatus yet:
