UNBOKS — Build simple shared Tasks page for Calvin + Jr

Goal:
Create a simple shared task page at:

/tasks

Also support:

/Tasks → redirect to /tasks

Purpose:
Calvin and Jr work remotely and need one simple place to post text tasks and screenshots for each other.
Nothing fancy.
One person posts a task, the other can read it.
When done, the task can be marked as Done.

Important:
This must be shared between different users/devices.
Do NOT use localStorage as the source of truth.
Use backend/API/shared storage.
If backend endpoints do not exist yet, build the frontend against a clean API contract and clearly mark the backend endpoints needed.

Page:
unboks.org/tasks

Access:
This page should not be public/open to everyone.
Use a very simple protection for v1:
- password gate, or
- existing admin/login mechanism if available
- allowed users: Calvin and Jr

Do not overbuild authentication.

Core UI:
1. Page title:
   Tasks

2. Subtitle:
   Shared task board for Calvin and Jr.

3. New task composer:
   - “To” selector:
     - Jr
     - Calvin
   - Rich text field
   - Screenshot/image paste support
   - Button: Add task

4. Screenshot/image support:
   Calvin and Jr must be able to paste screenshots directly into the task composer.

Required behavior:
- User can copy a screenshot and press Cmd+V / Ctrl+V into the composer.
- Pasted image appears as an attachment preview before saving.
- User can remove an attached screenshot before saving.
- User can attach more than one screenshot if easy; if not, support at least one.
- Saved task displays attached screenshots below the task text.
- Clicking a screenshot opens a larger preview/lightbox or opens the image in a new tab.
- Screenshots must persist after refresh and be visible to the other user.
- Do not store images only in localStorage.
- Images must be uploaded to backend/shared storage or stored through the chosen persistent storage mechanism.
- If upload/storage is not available yet, build the UI and define the API contract clearly, but do not fake persistence.

Supported image types:
- PNG
- JPG/JPEG
- WebP

Reasonable limits:
- Max 5 images per task
- Max 10 MB per image
- Show clear error if too large or unsupported.

5. Task list:
   Each task card shows:
   - task text, rich text rendered safely
   - attached screenshots/previews
   - created by
   - assigned to
   - created date/time
   - status: Open / Done
   - Done button
   - optional Reopen button if already done

6. Filters:
   - Open
   - Done
   - All

7. Sorting:
   - Open tasks first
   - newest first within each group

8. Mobile:
   Must be clean and usable on phone.
   Calvin must be able to quickly add a task from mobile.
   Mobile should support selecting/uploading screenshots from photo library if paste is not available.

Rich text:
Keep it simple.
Support:
- bold
- italic
- bullet list
- numbered list
- links
- line breaks

Do not build a huge Notion-style editor.
Use existing editor/component if already available.
If no rich text editor exists, use a textarea for v1 but preserve line breaks and links.

Task data model:
Task:
{
  id: string,
  bodyHtml: string,
  bodyText: string,
  createdBy: "Calvin" | "Jr",
  assignedTo: "Calvin" | "Jr",
  status: "open" | "done",
  attachments: [
    {
      id: string,
      fileName: string,
      mimeType: "image/png" | "image/jpeg" | "image/webp",
      sizeBytes: number,
      url: string,
      createdAt: string
    }
  ],
  createdAt: string,
  updatedAt: string,
  completedAt?: string,
  completedBy?: "Calvin" | "Jr"
}

API contract:
GET /api/unboks/tasks
Returns all tasks Calvin/Jr can see.

POST /api/unboks/tasks
Use multipart/form-data if screenshots are included.

Fields:
- assignedTo
- bodyHtml
- bodyText
- attachments[]

Alternative acceptable API:
1. POST /api/unboks/tasks/uploads
   Upload image(s), returns attachment URLs/IDs.
2. POST /api/unboks/tasks
   Creates task with attachment IDs.

PATCH /api/unboks/tasks/:id
Body examples:
{
  status: "done"
}

or:
{
  status: "open"
}

DELETE is not required for v1.
Editing existing tasks is not required for v1 unless very easy.

Important behavior:
- When Calvin adds a task for Jr, it appears for Jr.
- When Jr adds a task for Calvin, it appears for Calvin.
- When Calvin/Jr pastes screenshots, the other person can view them.
- When Jr marks it Done, Calvin sees it as Done.
- Done tasks should not disappear completely; they move to Done.
- No notifications required in v1.
- No comments required in v1.
- No deadlines required in v1.
- No priorities required in v1.
- No kanban board.
- No drag/drop.
- No AI.
- No overengineering.

Design:
Simple, clean, Unboks style.
White background.
Clear cards.
Big readable text.
Obvious Done button.
Attachment previews should be clean thumbnails.
Not fancy.

Copy:
Empty Open state:
“No open tasks.”

Empty Done state:
“No completed tasks yet.”

Add task placeholder:
“Write a task…”

Paste hint:
“Paste screenshots here or attach images.”

Done button:
“Mark done”

Reopen button:
“Reopen”

Safety:
- Sanitize rich text before rendering.
- Do not allow script tags or unsafe HTML.
- Validate image mime type and size.
- Do not expose secrets.
- Do not touch DNS.
- Do not touch existing website pages except routing/nav if needed.
- Do not break the public website.
- Do not touch dashboard frontend unless this page lives there intentionally.
- Do not modify unrelated backend systems.

Files likely involved:
- Public website router/App
- New Tasks page component
- API client helper
- Image upload handling
- Optional backend task endpoints if this repo owns them

Acceptance:
- /tasks loads.
- /Tasks redirects or opens same page.
- Calvin can add a text task for Jr.
- Jr can add a text task for Calvin.
- Calvin/Jr can paste a screenshot into the composer.
- Screenshot preview appears before saving.
- Saved task displays screenshot after refresh.
- Other user can see screenshot.
- Task can be marked Done.
- Done task appears in Done filter.
- Reopen works if implemented.
- Mobile layout works.
- No local-only fake storage as final source.
- Build clean.
