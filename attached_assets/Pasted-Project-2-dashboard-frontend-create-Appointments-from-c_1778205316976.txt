Product issue:
A soft escalation scheduling flow resulted in a confirmed appointment, but the appointment does not appear in Workspace → Appointments.

Conversation example:
Customer wanted to meet.
Customer gave availability:
“tomorrow 10-12 thursday 09.00-12:00 those are my times”

Operator replied to Marina choosing:
Thursday 09:00 at Café Paris in Willemstad

Marina replied:
“Thursday 09:00 at Café Paris in Willemstad, passed to the team. They'll confirm shortly. Marina”

This should be treated as an appointment.

Important product rule:
When customer availability and operator-selected time/location match, the system should create or show an appointment.

In this case:
- Customer offered Thursday 09:00 to 12:00.
- Operator chose Thursday 09:00.
- Operator added location Café Paris in Willemstad.
- Marina communicated that to the customer.
- Therefore this should show as an appointment.

Required behavior:
1. Appointments section must show confirmed or pending appointments created from scheduling conversations.
2. A conversation where:
   - customer asks to meet/book/schedule/activate
   - customer gives availability
   - operator/Marina confirms or selects a date/time/location
   should produce an appointment row/card.
3. Appointment should include:
   - Customer name
   - Channel
   - Date/time
   - Location
   - Appointment topic/purpose
   - Status
   - Source conversation link/open action
4. The appointment should appear under Workspace → Appointments.
5. The appointment should not remain hidden only inside the message trail.
6. If backend appointments endpoint is not available yet, frontend may show detected appointment candidates from conversation data as “Detected appointment” or “Pending sync”.
7. Do not pretend backend saved the appointment unless backend confirms it.
8. Do not create fake appointments from vague scheduling talk where no time was chosen.
9. Do not create duplicate appointments for the same conversation/time.
