Bug:
When trying to upload or paste an image/screenshot into a task, the request fails with:
- HTTP 422
or
- HTTP 401

Context:
The Tasks backend may not fully support image upload yet.
The frontend must handle this cleanly and must not break task creation.

Tasks page currently supports local pending tasks while backend is unavailable.
Image upload must follow the same principle:
- If backend upload is unavailable, store/show image locally if safe.
- If auth fails, show clear session/auth message.
- Do not crash.
- Do not lose the typed task text.

Task:
Audit and fix the Tasks image upload/paste flow.

Required investigation:
1. Find where pasted/attached images are handled:
   - TaskComposer.tsx
   - TaskCard.tsx
   - tasks-api.ts
   - any local pending tasks hook
2. Check whether image upload is sent to the correct endpoint.
3. Check whether Authorization: Bearer token is included if calling api.unboks.org.
4. Check whether the request body matches backend expectation:
   - multipart/form-data if uploading files
   - do NOT manually set Content-Type for FormData; browser must set boundary.
5. Check whether 422 is caused by wrong field name:
   - attachments
   - files
   - image
   - upload
6. Check whether 401 is caused by missing token or expired token.
7. Check whether image upload should be skipped entirely while task backend is unavailable.

Required behavior:
1. User can paste or attach an image without the page breaking.
2. If backend upload endpoint is unavailable or returns 404/422:
   - keep image as local pending attachment if feasible
   - show calm note:
     “Image saved locally. It will sync when task uploads are connected.”
3. If backend returns 401/403:
   - do not silently fail
   - show:
     “Session expired. Please sign in again.”
   - keep the unsaved task text and local image preview if possible.
4. If image is too large:
   - show:
     “Image is too large. Maximum size is 10 MB.”
5. If image type unsupported:
   - show:
     “Unsupported image type. Use PNG, JPG, or WebP.”
6. Task creation without image must still work.
7. Task creation with local pending image must work locally.
8. The image preview should appear before saving.
9. User can remove the image before saving.
10. Local pending task should show the image preview after saving and refresh if stored safely.
11. Do not fake that Jr can see local images before backend sync.
12. Clearly mark local-only image attachments as pending sync.

Important:
If the Python backend does not yet support image uploads, do not keep retrying and throwing red errors.
Fallback to local pending behavior.

Suggested implementation:
- Validate file type and size before upload.
- For local pending mode:
  - use object URL for immediate preview
  - for persistence across refresh, store small image data URLs only if size is safe
  - if image is too large for local persistence, keep preview for current session and show warning
- Max 5 images per task.
- Max 10 MB per image.
- Accepted:
  - image/png
  - image/jpeg
  - image/webp

API handling:
- If using FormData:
  const formData = new FormData()
  formData.append("file", file) or correct field expected by backend
  fetch(url, {
    method: "POST",
    headers: { Authorization: `Bearer ${token}` },
    body: formData
  })
- Do not set "Content-Type": "multipart/form-data" manually.
- Only use upload endpoint if it exists.
- If upload endpoint does not exist, do not block local task creation.
